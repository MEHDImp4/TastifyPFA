**Step 1.** Edit `backend/apps/commandes/views.py`. After the existing `def perform_create(self, serializer):` line and BEFORE `@action(detail=True, methods=['post']) def add_items(...)`, insert two new methods:

```python
    def update(self, request, *args, **kwargs):
        """Phase 16 (T-16-01): SERVEUR may only mutate orders they own; GERANT bypasses."""
        instance = self.get_object()
        if instance.serveur != request.user and request.user.role != 'GERANT':
            return Response(
                {"error": "Vous n'êtes pas autorisé à modifier cette commande."},
                status=status.HTTP_403_FORBIDDEN,
            )
        return super().update(request, *args, **kwargs)

    def partial_update(self, request, *args, **kwargs):
        """Phase 16 (T-16-01): same ownership rule as update() applies to PATCH."""
        instance = self.get_object()
        if instance.serveur != request.user and request.user.role != 'GERANT':
            return Response(
                {"error": "Vous n'êtes pas autorisé à modifier cette commande."},
                status=status.HTTP_403_FORBIDDEN,
            )
        return super().partial_update(request, *args, **kwargs)
```

These methods sit alongside `update()`/`partial_update()` inherited from `ModelViewSet`. They short-circuit BEFORE serializer validation, returning the standardized error message also used by `add_items` (lines 66-70) and `destroy` patterns.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated SERVEUR -> CommandeViewSet PATCH/PUT | A logged-in SERVEUR may attempt to mutate any commande PK. Without ownership guards on update/partial_update, the SERVEUR can: (a) fire another SERVEUR's order, (b) regress a PRETE order to EN_COURS, (c) cancel another SERVEUR's order. |
| Authenticated CUISINIER -> CommandeViewSet PATCH | A logged-in CUISINIER (role passes `IsCuisinierOrGerant` permission) attempting to PATCH a Commande would not own it. The new ownership guard rejects them too — kitchen-side state changes (e.g. PRETE) are deferred to a dedicated Phase 17+ endpoint. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Elevation of Privilege | `CommandeViewSet.partial_update` (and `.update`) | mitigate | Add ownership check `if instance.serveur != request.user and request.user.role != 'GERANT': return 403`. Locked by `test_fire_order_non_owner_serveur_forbidden`. |
| T-16-06 | Tampering | `CommandeViewSet.update/partial_update` (status regression PRETE -> EN_COURS) | accept (deferred to Phase 17+) | Documented in CONTEXT.md / RESEARCH.md Security Domain. Phase 17 will introduce a state-machine validator. Phase 16 prevents most regressions implicitly because only the owner can attempt them. |
