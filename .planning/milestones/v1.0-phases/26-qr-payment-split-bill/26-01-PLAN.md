# Phase 26 Plan 01: Payment Domain, Payable Session Resolution & Lifecycle Integrity

## Goal
Establish the backend payment domain and its non-UI invariants: persistent payment models, a single authoritative payable-session rule for table-to-order resolution, atomic split/lifecycle services, and backend validation that proves the order can transition to `PAYEE` safely without double-payment or table-state drift.

## Scope Guard
- Backend only. Do not add Salle modal work, client landing page UI, or any Phase 27 presentation concerns.
- This plan stops at domain services and payable-session resolution. Public payment API surfaces and QR session endpoints belong to Plan 26-02.
- Anonymous access is out of scope here; this plan prepares the domain/service layer that later token-authorized endpoints will call.

## Deliverables
- New Django app: `app/backend/apps/paiements`
- Payment domain models, migrations, app registration, and signal wiring
- Atomic service layer for split math, contribution validation, paid-total reconciliation, and order completion
- Explicit payable-session rule for mapping a table to exactly one payable `Commande`
- Validation tests for rounding, over-coverage prevention, ambiguous sessions, and `Commande -> PAYEE -> table freed` lifecycle

## Payable Session Rule
- A table is payable only when exactly one `Commande` for that table satisfies all of the following:
  - `est_active = True`
  - `statut IN (EN_COURS, EN_CUISINE, PRETE)`
  - `montant_total > sum(completed paiements)`
- Resolution must lock candidate commandes for the table in deterministic `id` order before deciding.
- If no candidate exists, the resolver returns a domain error for `no payable order`.
- If more than one candidate exists, the resolver returns a domain error for `ambiguous payable order` and no QR/payment flow may proceed until staff resolves the table state.
- `PAYEE` and `ANNULEE` commandes are never payable candidates, even if stale tokens later reference them.

### 2. Implement atomic split, contribution, and payable-session services
- **Files**:

  - `app/backend/apps/paiements/services.py`

- **Action**:
  - Model the service layer after `apps.stock.services` and `apps.reservations.services`: wrap state-changing flows in `transaction.atomic()`, use `select_for_update()`, and lock rows in deterministic order to avoid deadlock inversion.
  - Implement a payable-session resolver that locks the table’s candidate commandes, enforces the single-payable-order rule above, and returns a domain object the API layer can consume later.

  - Implement item/fraction split validation against each `CommandeLigne` payable amount using aggregated completed contributions plus the in-flight contribution request, rejecting any over-coverage before save.
  - Lock the parent `Commande` and targeted `CommandeLigne` rows before coverage checks so concurrent payers cannot double-pay the same line or exceed the remaining order balance.

## Risks & Guards
- **Concurrency**:
  - Guard with `transaction.atomic()` and `select_for_update()` on `Commande` plus targeted `CommandeLigne` rows.
  - Lock rows in deterministic primary-key order to avoid deadlocks.
- **Ambiguous table state**:
  - Treat multiple non-terminal commandes on one table as a blocking domain error, not a `pick the latest` fallback.
- **Rounding drift**:
  - Centralize split rounding rules in the payment service and treat the final share as the balancing share.
- **Lifecycle duplication**:
  - Payment completion updates only the order payment state; table release continues through existing `apps.commandes.signals`.

## Done When
- `apps.paiements` exists, is registered, and migrates cleanly.
- The project has one authoritative payable-session resolver with explicit no-order and ambiguous-order failures.
- Equal split and item/fraction split calculations are implemented and tested at the domain layer.
- Payment completion can drive `Commande.statut = PAYEE` without bypassing existing table-sync behavior.
