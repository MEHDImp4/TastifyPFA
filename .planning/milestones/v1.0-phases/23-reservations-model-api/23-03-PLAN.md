<objective>
Close two blocking gaps and two advisory issues surfaced by the Phase 23 verifier and code reviewer.

Gap 1 (BLOCKER — CR-01): `validate_statut` in `reservations/serializers.py` only guards updates (`self.instance is not None`). On POST (create) `self.instance` is `None`, so a client can supply `{"statut": "PRESENTE"}` and the reservation persists with a staff-only status. Fix: make the guard unconditional — non-staff users cannot supply any statut on create.

Gap 2 (BLOCKER — CR-02): `_compute_statut_effectif` in `tables/serializers.py` converts the buffered end-time to `.time()` before comparing. When `heure_fin + 15 min` crosses midnight (e.g., 23:55 → 00:10), the `.time()` value wraps below `heure_debut` and the condition is always false after midnight — an occupied table appears free. Fix: compare full `datetime.datetime` objects throughout.

Advisory WR-03: `_today_reservations_prefetch` in `tables/views.py` populates `table._today_reservations` via `to_attr`, but `_compute_statut_effectif` ignores it and issues a fresh queryset per table (N+1). Fix: read the prefetch attribute when it exists; fall back to a DB query only when absent.

Advisory CR-03: `update_reservation` in `services.py` acquires the old-table lock then the new-table lock without ordering. Two concurrent updates swapping tables can deadlock. Fix: lock both PKs together in ascending order.

Purpose: Bring Phase 23 to full goal achievement — 6/6 must-have truths verified with no blockers.
Output: Four targeted fixes across four files, each independently verifiable.

    """
    Derives reservation-aware table status at read time (T-23-06).

    A table is treated as RESERVEE when it has an active reservation whose
    buffered window (heure_fin + RESERVATION_CLEANUP_BUFFER) overlaps with now.
    The stored Table.statut is never mutated; this is a pure derived field.

    Reads from the prefetch attribute _today_reservations when available (WR-03)
    to avoid N+1 queries on the list endpoint. Falls back to a DB query only
    when the attribute is absent (e.g., retrieve endpoint without prefetch).

    Compares full datetime objects to handle midnight-straddling windows
    correctly (e.g., heure_fin=23:55 + 15 min buffer = 00:10 next day) — CR-02.
    """
    from apps.reservations.constants import RESERVATION_CLEANUP_BUFFER
    from apps.reservations.models import Reservation

    now = datetime.datetime.now()
    today = now.date()

    if hasattr(table, '_today_reservations'):
        active_today = table._today_reservations
    else:
        active_today = (
            Reservation.objects.active()
            .filter(table_id=table.pk, date_reservation=today)
        )

        start_dt = datetime.datetime.combine(today, reservation.heure_debut)

            datetime.datetime.combine(today, reservation.heure_fin)

Replace `update_reservation` (lines 69-83) with a version that locks both table PKs in ascending order before acquiring any row lock.

Current code with deadlock risk:
```python
def update_reservation(reservation, **changes):
    with transaction.atomic():
        locked_table = Table.objects.select_for_update().get(pk=reservation.table_id)
        locked_reservation = Reservation.objects.select_for_update().get(pk=reservation.pk)
        if 'table' in changes:
            locked_table = Table.objects.select_for_update().get(pk=changes['table'].pk)
            locked_reservation.table = locked_table
        for field, value in changes.items():
            if field == 'table':
                continue
            setattr(locked_reservation, field, value)
        locked_reservation.table = locked_table
        locked_reservation.save()
        return locked_reservation
```

Replace with:
```python
def update_reservation(reservation, **changes):
    with transaction.atomic():
        old_table_pk = reservation.table_id
        new_table_pk = changes['table'].pk if 'table' in changes else old_table_pk

        # Lock both table rows in ascending PK order to prevent deadlock inversion
        # when two concurrent transactions swap tables in opposite directions (CR-03).
        pks_to_lock = sorted({old_table_pk, new_table_pk})
        Table.objects.select_for_update().filter(pk__in=pks_to_lock)

        locked_reservation = Reservation.objects.select_for_update().get(pk=reservation.pk)
        for field, value in changes.items():
            setattr(locked_reservation, field, value)
        locked_reservation.save()
        return locked_reservation
```

Key changes:
- Both PKs collected into a set, sorted ascending, then locked in a single `filter(pk__in=...)` call — database processes them in index order, eliminating lock inversion.
- The redundant double-lock pattern (get old, then get new, overwriting the variable) is replaced with a single `filter`.
- The explicit `locked_reservation.table = locked_table` line (which was unconditional and caused confusion when table was not in changes) is removed — `setattr(locked_reservation, field, value)` in the loop already handles `'table'` correctly when it is in changes.
- `for field, value in changes.items()` no longer skips `'table'` — `setattr` with the Table object from `changes['table']` is safe and correct.

Do NOT change `create_reservation`, `is_table_available`, or any other function.
