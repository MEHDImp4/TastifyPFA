<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer machine ↔ git remote | Secrets in `.env` must never cross this boundary |
| filesystem ↔ docker container | `.env` is mounted/read by docker-compose; values become process env in containers |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Information Disclosure | `.env` accidentally committed to git | mitigate | `.gitignore` line `^\.env$` (Task 1); `.env.example` committed instead with placeholders only (Task 2). Acceptance criterion explicitly verifies `git ls-files .env` returns empty. |
| T-1-02 | Information Disclosure | `SECRET_KEY` placeholder remains in production | mitigate | `.env.example` ships with `replace-me-...-cli` placeholder making it obvious; `dev.py` settings (Plan 02) require `config('SECRET_KEY')` which raises if unset. ASVS V6.4.1. |
| T-1-03 | Tampering | `.env` writable by other local users | accept | Single-developer machine (per CLAUDE.md project context); not a production threat surface in Phase 1. |
| T-1-04 | Information Disclosure | README.md or FILE_MAP.md leaks secret values | mitigate | Task 3 explicitly forbids real secret values in README/FILE_MAP — only structural placeholders. |
</threat_model>
