<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Backend container -> Redis broker | Internal Docker network; broker accepts task payloads from any service on tastify-network. |
| Celery worker -> Django ORM | Worker process loads Django apps and accesses MySQL with same privileges as backend. |
| docker-compose.yml -> runtime | Service definitions become privileged Linux containers when started. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | D (DoS) | celery-worker concurrency | mitigate | Hard-cap `--concurrency=2` in docker-compose so a runaway scheduler cannot fork unbounded workers. |
| T-15-02 | D (DoS) | Celery task runtime | mitigate | Set `CELERY_TASK_TIME_LIMIT=60` and `CELERY_TASK_SOFT_TIME_LIMIT=30` so a malformed broadcast cannot block a worker indefinitely. |
| T-15-03 | T (Tampering) | Redis broker network | mitigate | Redis is private to `tastify-network`; not exposed via `ports:`. Documented in research as standard mitigation for Task Injection. |
| T-15-04 | I (Info Disclosure) | CELERY_BROKER_URL secret | accept | Default value uses internal hostname `redis://redis:6379/0`; production overrides via .env. No credentials in repo. |
</threat_model>
