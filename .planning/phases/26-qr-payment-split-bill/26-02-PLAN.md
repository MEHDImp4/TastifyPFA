# Phase 26 Plan 02: QR Token Authorization & Payment API Contracts

## Goal
Expose the Phase 26 payment flows through explicit, execution-ready backend APIs: staff QR issuance, token-backed session resolution, preview-only split calculators, and mutating payment-recording endpoints with a fixed contract for self-service and staff-cash settlement.

## Scope Guard
- Backend only. Do not add the staff encaissement modal, client QR landing page UI, or any Phase 27 rendering/state concerns.
- This plan consumes the domain and payable-session services created in Plan 26-01; do not re-decide split math or lifecycle rules here.
- Keep token-backed client access scoped to dedicated endpoints only. Do not relax global DRF authentication defaults.

## API Contract

### Staff QR issuance
- `GET /api/tables/{id}/qr/`
  - Staff-only (`SERVEUR` and `GERANT`).
  - Resolves the payable session for the table through the Plan 26-01 service.
  - Returns `404` when no payable order exists.
  - Returns `409` when multiple payable candidates exist for the table.
  - Returns a signed payload or URL containing `table_id` and `commande_id`.

### Token-backed session preview
- `POST /api/paiements/session/resolve/`
  - Token-authorized for client self-service; staff may also call it for diagnostics.
  - Read-only. No `Paiement` row is created here.
  - Validates signature, `max_age`, `table_id`, `commande_id`, and current payable session state.
  - Returns order snapshot, remaining balance, and per-line contribution state.

### Preview-only split calculators
- `POST /api/paiements/session/equal-split/`
  - Token-authorized.
  - Read-only. No `Paiement` row is created here.
  - Accepts `split_count` and optional `shares_to_pay_now`.
  - Returns normalized share amounts with the last share absorbing rounding residue.
- `POST /api/paiements/session/item-split/`
  - Token-authorized.
  - Read-only. No `Paiement` row is created here.
  - Accepts line-level contribution requests and returns validated allocations or field errors.

### Mutating payment creation
- `POST /api/paiements/session/pay/`
  - Token-authorized self-service endpoint.
  - Creates `Paiement` plus any `PaiementItem` rows after re-running the same validation under lock.
  - Requires `methode = EN_LIGNE`.
  - Requires `reference_transaction`.
  - Writes `statut = COMPLETE` only when the request is internally consistent and the reference is present; `EN_ATTENTE` is reserved for future provider callback flows and should not be produced by the Phase 26 self-service path.
- `POST /api/paiements/`
  - Staff-authenticated manual entry path.
  - Allows `methode = ESPECES` or `CARTE`.
  - `reference_transaction` is optional for `ESPECES` and required for `CARTE` if implementation chooses to capture card references manually.
  - Creates `Paiement` and `PaiementItem` rows through the same locked service path used by the token-backed endpoint.

  - Preview endpoints never create payment rows.
  - `session/pay` and staff manual payment both create payments through the same locked validation path.

## Risks & Guards
- **Contract drift**:
  - Freeze preview-only versus mutating endpoint responsibilities before execution; do not let views invent behavior ad hoc.
- **Authorization leakage**:
  - Keep token-backed access isolated to dedicated views and permissions instead of weakening global auth rules.
- **Stale session reuse**:
  - Every token-backed endpoint must re-resolve the payable session rather than trusting the token payload alone.
- **Phase leakage**:
  - No frontend routes, modal flows, QR rendering, or client page behavior belongs in this plan.

## Done When
- Staff can request a QR token only for a table with exactly one payable order.
- Token-backed preview and payment endpoints are implemented with fixed, documented responsibilities.
- The API test suite names and covers permission, no-order, ambiguous-order, stale-token, and payment-completion cases explicitly.
