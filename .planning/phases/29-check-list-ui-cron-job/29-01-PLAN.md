# Phase 29-01 Plan: Check-list UI & Cron Job

**Date:** 2026-05-07  
**Status:** Ready

## Outcome

Ship the backoffice checklist module with a dedicated route, inline execution completion, GERANT-only template management, and automated frontend verification.

## Work Items

1. Add route and navigation wiring
- Register `/checklists` in `app/frontend/backoffice/src/App.tsx`.
- Add the sidebar item in `app/frontend/backoffice/src/components/layout/Sidebar.tsx`.

2. Add checklist client layer
- Create typed checklist models for templates, executions, responses, and payloads.
- Create a service module for list, create, update, and response patch operations.

3. Build the page
- Add `app/frontend/backoffice/src/pages/Checklists/index.tsx`.
- Render a top summary area with today's date, counts, progress, and refresh.
- Render execution cards with inline response toggles.
- Add local optimistic update handling with rollback on failure.
- Add empty, loading, and error states.

4. Add GERANT management surfaces
- Add drawer components for template create/edit and manual execution creation.
- Keep management controls inside the same module and hide them for non-GERANT users.

5. Add tests
- Cover route/page rendering, execution loading, inline completion, and GERANT-only controls.
- Mock `axiosInstance` consistently with existing Vitest patterns.

6. Validate and sync
- Run the relevant frontend tests.
- Run `npm run build` for the backoffice app.
- Sync planning/docs/dashboard and commit focused phase changes.

## Risks

- The checklist API returns nested structures, so optimistic updates must preserve nested response/task references without accidental mutation.
- Template editing replaces task lists wholesale on the backend, so the form should submit a complete ordered task array every time.

## Verification Target

- Staff users can complete today's checklist items without page reloads.
- GERANT users can manage templates and create manual executions from the same page.
- The SPA builds cleanly and the new tests pass.
