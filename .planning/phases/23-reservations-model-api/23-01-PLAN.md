<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client input -> reservation domain | Date/time/party-size fields are fully user-controlled and must be validated before persistence. |
| concurrent request -> database rows | Two requests may target the same table and slot at the same time, so validation alone is insufficient without locking. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-01 | Tampering | `Reservation.clean()` and service create path | mitigate | Validate time ordering, capacity, and active-overlap rules before save; centralize the 15-minute buffer so every write path uses the same rule. |
| T-23-02 | Elevation of Privilege | Reservation `client` linkage | mitigate | In this plan, keep domain creation APIs service-based so Plan 23-02 can inject `request.user` rather than trusting payload-supplied client IDs. |
| T-23-03 | Denial of Service | Concurrent double-booking | mitigate | Use `transaction.atomic` plus `select_for_update()` in the reservation service to serialize conflicting writes for the same table slot. |
</threat_model>
