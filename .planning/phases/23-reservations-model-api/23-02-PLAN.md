<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated user -> reservation API | Role and ownership determine whether the caller may see or mutate reservation records. |
| reservation API -> table API representation | Reservation data influences what tables look available, so derived status must use the same validated service logic as writes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-04 | Information Disclosure | `ReservationViewSet.get_queryset()` | mitigate | Scope client users to `client=request.user` and reserve global visibility for `GERANT` / `SERVEUR` only. |
| T-23-05 | Elevation of Privilege | serializer status updates | mitigate | Enforce that clients may only transition to `ANNULEE`; only staff may set operational statuses such as `CONFIRMEE`, `PRESENTE`, or `ABSENTE`. |
| T-23-06 | Tampering | table status exposure | mitigate | Compute derived table reservation status from the shared reservation service so clients cannot bypass the 15-minute buffer by reading a weaker code path. |
</threat_model>
